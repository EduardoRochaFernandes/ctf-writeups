## Overview

The Security Information and Event Management (SIEM) platform is the operational centre of a SOC. It aggregates logs from every source in the environment, normalises them into a consistent format, applies detection rules to identify suspicious patterns, and presents alerts and raw data through an analyst interface. This room covers what a SIEM does, how it ingests data, how detection rules work, and the event IDs that matter most for blue team detection.

---

## What a SIEM Does

Three core functions:

1. **Collection and storage** — receives logs from every source (servers, endpoints, network devices, applications, cloud platforms) and stores them in a searchable index
2. **Normalisation and correlation** — standardises field names across log sources so that a single query or rule can span multiple source types simultaneously
3. **Alerting** — evaluates incoming logs against detection rules; when conditions are met, creates an alert for the analyst queue

Popular SIEM platforms: **Splunk**, **Elastic Security**, **Microsoft Sentinel**, **IBM QRadar**, **Sumo Logic**.

## Log Sources

**Host-Centric — events generated by individual systems:**

*Windows Event Logs:*
- Stored in `C:\Windows\System32\winevt\Logs\` as `.evtx` files
- Viewable with Event Viewer or shipped via Winlogbeat/Wazuh Agent
- Each event type has a unique Event ID

*Linux System Logs:*
- `/var/log/auth.log` or `/var/log/secure` — authentication events
- `/var/log/syslog` or `/var/log/messages` — general system events
- `/var/log/kern.log` — kernel events
- Application logs: `/var/log/nginx/`, `/var/log/apache2/`, etc.

**Network-Centric — events from devices monitoring traffic:**
- Firewall allow/deny logs
- IDS/IPS alert logs
- Web proxy access logs
- VPN authentication logs
- DNS query logs

## Critical Windows Event IDs

| Event ID | Description | Why It Matters |
|----------|-------------|----------------|
| 4624 | Successful logon | Baseline; anomalies indicate compromised credentials |
| 4625 | Failed logon | Brute force indicator when seen in volume |
| 4688 | Process creation (with command line if auditing enabled) | Core endpoint telemetry for execution detection |
| 4698 | Scheduled task created | Common persistence mechanism |
| 4720 | User account created | Potential persistence or privilege escalation |
| 4728/4732 | Member added to privileged/local group | Privilege escalation indicator |
| 1102 | Security audit log cleared | Defence evasion — attacker removing evidence |
| 104 | System log cleared | Same significance as 1102 |
| 1116 | Windows Defender detected malware | Immediate investigation required |

## Log Ingestion Methods

| Method | Description |
|--------|-------------|
| Agent/Forwarder | Lightweight software on endpoint that ships logs in real time. Splunk calls this the Universal Forwarder; Elastic uses Filebeat/Winlogbeat. |
| Syslog | Standard protocol for real-time log shipping. Widely supported by network devices, servers, and appliances. |
| API | Cloud platforms and SaaS applications expose log data via API. SIEM polls or subscribes via webhooks. |
| Manual Upload | Import historical log files offline for retrospective analysis. Useful during IR when analysing pre-breach activity. |

## How Detection Rules Work

A detection rule is a logical expression evaluated against incoming or indexed log data. When the conditions are met, an alert is created.

**Examples:**

| Rule Condition | Alert Generated |
|---------------|----------------|
| `EventID = 1102` | Security log cleared — attacker covering tracks |
| `EventID = 4688 AND NewProcessName CONTAINS 'whoami'` | Post-exploitation enumeration |
| `Failed logins > 5 in 30 seconds from same source IP` | Brute force attempt |
| `Outbound data volume > 500MB to external IP` | Potential data exfiltration |
| `USB device inserted on restricted asset` | Policy violation |

**Why normalisation is critical:** If Windows calls the username field `TargetUserName` and Linux syslog uses `user`, a single rule cannot span both sources without normalisation. Normalisation maps source-specific field names to a common schema so that `user` means the same thing regardless of which system generated the log.

## SIEM vs EDR vs NDR

A common question is how these three platforms relate to each other:

| Platform | Primary data | Coverage |
|---------|-------------|---------|
| SIEM | All log sources | Correlation across entire environment |
| EDR | Endpoint telemetry | Deep endpoint visibility, response |
| NDR | Network traffic | Network-level detection, lateral movement |

The SIEM is the aggregation and correlation layer. EDR and NDR feed their alerts and telemetry into the SIEM, which correlates them with each other and with logs from other sources to identify attack chains that no single platform would detect in isolation.

---

## Key Takeaways

The SIEM is only as valuable as the quality of the data feeding it and the quality of the rules applied to that data. A SIEM receiving logs from only half the environment will miss attacks that originate or persist in the unmonitored half. Detection rules that fire on normal behaviour create alert fatigue that masks real threats. Getting both right — comprehensive coverage and low false positive rates — is the ongoing challenge of detection engineering.
