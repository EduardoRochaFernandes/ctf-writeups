## What is this room about?

DoS and DDoS attacks aim to make a web application unavailable by overwhelming its resources. This room covers the types of web-layer attacks, how to detect them in logs and with SIEM, and the technical and infrastructure defences available.

---

## DoS vs DDoS

- **DoS** — single machine, limited by one attacker's resources
- **DDoS** — botnet of thousands of compromised devices, coordinated to amplify the attack massively

---

## Types of Web Application DoS Attacks

| Attack | Mechanism |
|--------|-----------|
| **Slowloris** | Sends many incomplete HTTP requests simultaneously — ties up server connections without sending data |
| **HTTP Flood** | Massive volume of legitimate-looking HTTP requests |
| **Cache Bypass** | Appends random parameters (`?a=xyz`) to URLs — forces origin server to respond instead of CDN cache |
| **Oversized Query** | Requests with huge parameters (`?limit=999999`) that force expensive database queries |
| **Login/Form Abuse** | Floods authentication endpoints — triggers DB queries and session creation on every attempt |
| **Faulty Input Validation** | Exploits poor input handling to cause crashes or hangs |

---

## Detection in Access Logs

| Indicator | Why suspicious |
|-----------|---------------|
| High request rate from same IP | >1000 requests in seconds = not human |
| Unusual User-Agents | `curl/7.x`, `Python-urllib` — automated tools |
| Geographic anomalies | Traffic from dozens of countries simultaneously |
| Same-second bursts | Requests with identical timestamps = bot |
| Spike in 5xx errors | Server failing under load |
| `?limit=999999` style queries | Resource exhaustion attempt |

**Most targeted endpoints:**
- `/login` — triggers auth + DB query
- `/search` — complex DB queries
- `/api` — dynamic content generation
- `/register` — writes to DB + email sending
- `/checkout` — inventory + payment processing

## Detection with SIEM (Splunk Example)

```splunk
# Find IPs making unusually high request rates
index=web sourcetype=access_log
| bin _time span=1m
| stats count by _time, clientip
| where count > 100
| sort - count

# Find requests to login endpoint
index=web uri_path="/login.php"
| timechart count by clientip

# Identify automated user agents
index=web
| stats count by useragent
| where match(useragent, "curl|python|bot|crawler")
```

## Prevention and Mitigation

### Application Layer
- **Input Validation** — limit size, format, and content of all inputs at every endpoint
- **CAPTCHA** — blocks automated submissions at login, registration, contact forms
- **JavaScript Challenges** — verify human interaction transparently without visible CAPTCHA
- **Rate Limiting** — server-side: max N requests per minute per IP for sensitive endpoints

### Infrastructure Layer

**CDN (Content Delivery Network):**
- Serves cached content from edge servers close to users — origin server handles far fewer requests
- Load balancing distributes traffic across multiple servers
- Absorbs large DDoS volumes before they reach the origin
- Provides geographic visibility — breakdown of traffic by source country

**WAF (Web Application Firewall) integrated in CDN:**
- Inspects and blocks malicious requests based on rules
- Rate limiting per IP, per endpoint
- Integrates threat intelligence feeds
- Custom rules for specific attack patterns

---

## Real-World Scale

- **Google (2023)** — mitigated a DDoS peaking at 398 million requests per second
- **Cloudflare** — mitigated the largest recorded DDoS at 11.5 Tbps over 35 seconds

---

## How Attackers Evade Defences

- Cache bypass (`/products?rand=abc`) — forces origin to respond
- Rotating User-Agents — evade User-Agent-based WAF rules
- Spoofed Referer headers — appear to be legitimate referral traffic
- Geographically distributed botnets — evade geo-blocking

---

## Key Takeaways

> The CDN is your first line of defence against DDoS — it absorbs volume before it reaches your application. Rate limiting at the WAF is your second line.

The Cache Bypass technique was new to me — the idea that adding a random query parameter completely bypasses CDN caching is a simple but elegant attack. It means even content that should be fully cached has to be regenerated by the origin server for every request.
